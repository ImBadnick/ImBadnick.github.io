WORTH is the abbreviation for WORkTogetHer and is a project done for an university exam. The goal of this educational project is the organization and management of projects in a collaborative way. The project has been done in Java.

PROBLEM DESCRIPTION

In recent years, numerous collaborative applications have been created, for sharing content, messaging, video conferencing, project management, etc. In this educational project, WORTH (WORkTogetHer), we will focus on organizing and managing projects in a collaborative way. The collaboration and project management applications (eg Trello, Asana) help people get organized and coordinate in carrying out common projects. These can be professional projects, or in in general, any activity can be organized into a series of tasks (eg to do list) which are carried out by members of a group: the applications of interest are of different types, think of the organization of a software development project with colleagues from the development team, but also to organize a party with a group of friends.
Some of these tools (eg Trello) implement the Kanban method (sign or billboard, in Japanese), an “agile” management method. The Kanban board provides an overview of the activities e displays its evolution, for example from creation and subsequent progress to completion, after the review phase has been successfully passed. One person from the work group can take in charge of an activity when he has the opportunity, by moving the activity on the board.
The project consists in the implementation of WORkTogetHer (WORTH): a tool for managing collaborative projects inspired by some principles of the Kanban methodology.

SPECIFICATION OF OPERATIONS

Users can access WORTH after registration and login. In WORTH, a project, identified by a unique name, consists of a series of "cards" ("cards"), which they represent the tasks to be carried out in order to complete it, and it provides a series of services. For every project is associated with a list of members, that is users who have the permissions to modify the cards and access the services associated with the project (eg chat).
A card consists of a name and a text description. The name assigned to the card must be unique within a project. Each project has four associated lists that define the flow of work as a passage of cards from one list to the next: TODO, INPROGRESS, TOBEREVISED, DONE. Any member of the project can move the card from one list to another.
The newly created cards are automatically added to the TODO list. Any member can move one card from one list to another. When all the cards are in the DONE list the project can be deleted, from any member participating in the project.
A group chat is associated with each project, and all members of that project, if online (after having logged in), they can receive and send messages on the chat. On the chat the system also sends automatically notifications of events related to the movement of a project card from a list to the other.
Un utente registrato e dopo login eseguita con successo ha i permessi per:

  • Retrieve the list of all users registered to the service;
  • Retrieve the list of all users registered for the service and connected to the service (in online status);
  • Create a project;
  • Retrieve the list of projects of which he is a member.

A user who has created a project automatically becomes a member of it. Can add other users register as a member of the project. All project members have the same rights (the creator himself is a member like the others), in particular:

  • add other users registered as project members;
  • retrieve the list of project members;
  • create cards in the project;
  • retrieve the list of cards associated with a project;
  • retrieve the information of a specific card of the project;
  • recover the “history” of a specific card of the project (see below for details);
  • move any card of the project
  • send a message on the project chat;
  • read messages from the group chat;
  • delete the project.

The operations offered by the service are specified below. During implementation it is possible add additional parameters as needed. register (username, password): to insert a new user, the server provides an operation registration of a user. The server responds with a code that can indicate successful registration, or, if the nickname is already present, or if the password is empty, it returns an error message. How specified below, the records are among the information to be persisted.

login (username, password): login of an already registered user to access the service. The server responds with a code that can indicate successful login, or, if the user has already logged in or the password is incorrect, returns an error message.

logout (nickUser): logs the user out of the service.

listUsers(): used by a user to view the list of user nicknames registered to the service and their status (online or offline).

listOnlineusers(): used by a user to view the list of User nicknames registered to the service and online in that moment.

listProjects(): operation to retrieve the list of projects of which the user is a member.

createProject (projectName): operation to request the creation of a new project. If the operation is successful, the project is created and has as a member the user who requested its creation.

addMember (projectName, user nick): operation to add the user nickUser to the project projectname. If the user is registered, the addition as a member is performed without asking for consent a nickUser, if the user is not registered, the operation cannot be completed and the service returns a error message.

showMembers (projectName): operation to retrieve the list of project members.

showCards (projectName): operation to retrieve the list of cards associated with a project projectName.

showCard (projectName, cardName): operation to retrieve information (name, description textual, list in which it is found at that moment) of the card cardName associated with a projectName project.

addCard (projectName, cardName, description): operation to request the addition of the name card cardName to the project projectname. The card must be accompanied by a short descriptive text. There card is automatically added to the TODO list.

moveCard (projectName, cardName, listPartenza, listDestinazione): operation to request the move the card named cardName to the project projectname from the list Startlist to the list listDestination.

getCardHistory (projectName, cardName): operation to request the "history" of the card, that is sequence of events for moving the card, from creation to the most recent move.

readChat (projectName): operation to view projectName project chat messages.

sendChatMsg (projectName, message): the user sends a message to the projectName project chat cancelProject (projectName): A project member asks to cancel a project. The operation can be completed successfully only if all cards are in the DONE list.
